Points-to Analysis in LLVM:
What SVF Gets Right

Eshaan Pareek April 2026 8 min read

Static analysis is one of those fields where the gap between theory and practice is enormous. You can prove beautiful things about pointer semantics on paper, and then watch a real compiler choke on a codebase with ten thousand indirect calls. This post is about that gap — specifically as I encountered it while building a points-to validator for LLVM at the PLATO lab.

What is Points-to Analysis?

At its core, points-to analysis asks a simple question: at this program point, what memory locations might this pointer refer to? The answer lets a compiler eliminate redundant loads, prove memory safety, devirtualize indirect calls, and a dozen other high-value transformations.

The hardest part isn't computing the points-to sets. It's deciding what precision you can afford.

The classic taxonomy splits analyses along two axes: flow sensitivity (does the analysis track program order?) and context sensitivity (does it distinguish different call sites of the same function?). A flow-insensitive, context-insensitive analysis is fast but coarse. A fully flow- and context-sensitive analysis is precise but combinatorially explosive.

Where SVF Fits

SVF is a framework for static value-flow analysis built on top of LLVM IR. It implements Andersen-style inclusion-based points-to analysis, which sits in the middle of the precision-performance tradeoff. 

; A simple store through a pointer — what does %p point to?
%p = alloca i32*, align 8
%x = alloca i32, align 4
store i32* %x, i32** %p, align 8
%loaded = load i32*, i32** %p, align 8

For this snippet, Andersen's algorithm would correctly compute that %loaded points to %x. The inclusion constraint pts(%p) ⊇ {%x} propagates through the load to give pts(%loaded) = pts(%p).

The cycle problem

Where SVF — and Andersen analyses generally — struggle is with strong updates and recursive data structures. When pointers form cycles in the constraint graph, the analysis can produce spuriously large points-to sets that collapse all heap objects of the same type into one.

⚡ Key Insight
Cycle detection (Offline Variable Substitution / HVN / HRU) is what separates fast SVF runs from slow ones. If your analysis is taking hours, look at the SCC structure of your constraint graph first.

Building a Validator

The validator I built at PLATO works by running SVF on a compiled LLVM module and then independently checking a subset of its conclusions against a reference dynamic analysis. The dynamic trace provides ground truth: if a pointer actually points to a location at runtime, the static analysis must include that location in the points-to set (soundness).

  • Instrument the LLVM IR with shadow memory tracking at every load/store
  • Run the instrumented binary on a representative test suite
  • Diff the dynamic points-to sets against SVF's static sets
  • Flag any dynamic pointer target absent from the static set as a soundness violation

In practice, most "violations" turned out to be instrumentation artifacts — particularly around memcpy and va_arg intrinsics that SVF models conservatively. But a handful were genuine bugs in SVF's handling of certain GEP (GetElementPtr) patterns.
